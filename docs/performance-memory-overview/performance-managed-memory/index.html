Managed memory

Unity‚Äôs managed memory system is a C# scripting environment based on the Mono or IL2CPP Virtual Machines (VMs). The benefit of the managed memory system is that it manages the release of memory, so you don‚Äôt need to manually request the release of memory through your code.

Unity‚Äôs managed memory system uses a garbage collector and a managed heap to automatically free memory allocations when your scripts no longer hold any references to those allocations. This helps safeguard against memory leaks. Memory leaks occur when memory is allocated, the reference to it is lost, and then the memory is never freed because it needs a reference to it to free it.

This memory management system also guards memory access, which means that you can‚Äôt access memory that has been freed, or that was never valid for your code to access. However, this memory management process impacts runtime performance, because allocating managed memory is time-consuming for the CPU. Garbage collection might also stop the CPU from doing other work until it completes.

Value and reference types‚Äã

When a method is called, the scripting back end copies the values of its parameters to an area of memory reserved for that specific call, in a data structure called a call stack . The scripting back end can quickly copy data types that occupy a few bytes. However, it‚Äôs common for objects, strings, and arrays to be much larger, and it‚Äôs inefficient for the scripting back end to copy these types of data on a regular basis.

All non-null reference-type objects and all boxed value-typed objects in managed code must be allocated on the managed heap.

It‚Äôs important that you are familiar with value and reference types, so that you can effectively manage your code. For more information, see Microsoft‚Äôs documentation on value types, and reference types.

Automatic memory management‚Äã

When an object is created, Unity allocates the memory required to store it from a central pool called the heap , which is a section of memory that your Unity project‚Äôs selected scripting runtime (Mono or IL2CPP) automatically manages. When an object is no longer in use, the memory it once occupied can be reclaimed and used for something else.

Unity‚Äôs scripting back ends use a garbage collector to automatically manage your application‚Äôs memory, so that you don‚Äôt need to allocate and release these blocks of memory with explicit method calls. Automatic memory management requires less coding effort than explicit allocation/release and reduces the potential for memory leaks.

Memory fragmentation and heap expansion‚Äã

A quantity of memory, with some objects released represented by grey dashed lines.

The above diagram shows an example of memory fragmentation. When Unity releases an object, the memory that the object occupied is freed up. However, the free space doesn‚Äôt become part of a single large pool of ‚Äúfree memory.‚Äù

The objects on either side of the released object might still be in use. Because of this, the freed space is a ‚Äúgap‚Äù between other segments of memory. Unity can only use this gap to store data of identical or lesser size than the released object.

This situation is called memory fragmentation . This happens when there is a large amount of memory available in the heap, but it is only available in the ‚Äúgaps‚Äù between objects. This means that even though there is enough total space for a large memory allocation, the managed heap can‚Äôt find a large enough single block of contiguous memory to assign to the allocation.

The object annotated A, is the new object needed to be added to the heap. The items annotated B are the memory space that the released objects took up, plus the free, unreserved memory. Even though there is enough total free space, because there isn‚Äôt enough contiguous space, the memory for the new object annotated A can‚Äôt fit on the heap, and the garbage collector must run.

If a large object is allocated and there is insufficient contiguous free space to accommodate it, as illustrated above, the Unity memory manager performs two operations:

  • First, the garbage collector runs, if it hasn‚Äôt already done so. This attempts to free up enough space to fulfill the allocation request.
  • If, after the garbage collector runs, there is still not enough contiguous space to fit the requested amount of memory, the heap must expand. The specific amount that the heap expands is platform-dependent; however, on most platforms, when the heap expands, it expands by double the amount of the previous expansion.

Managed heap expansion considerations‚Äã

The unexpected expansion of the heap can be problematic. Unity‚Äôs garbage collection strategy tends to fragment memory more often. You should be aware of the following:

  • Unity doesn‚Äôt release the memory allocated to the managed heap when it expands regularly; instead, it retains the expanded heap, even if a large section of it is empty. This is to prevent the need to re-expand the heap if further large allocations occur.
  • On most platforms, Unity eventually releases the memory that the empty portions of the managed heap uses back to the operating system. The interval at which this happens isn‚Äôt guaranteed and is unreliable.
